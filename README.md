# Buy The Look

מערכת המלצות אופנה חכמה שמשלבת לוגיקה דטרמיניסטית עם AI שנותן הסבר אישי לכל המלצה. המטרה היא להציע למשתמש פריטים שבאמת מתאימים לו — לפי האירוע, התקציב, הסגנון והצבעים שהוא אוהב.

---

## גישה כללית לפתרון

בניתי מנוע שעובד בשני שלבים:

**שלב 1 — לוגיקה דטרמיניסטית:**
כל פריט מקבל ניקוד מחושב המבוסס על התאמה לסגנון, צבעים ואירוע, תוך סינון תקציבי וסינון צבעים אסורים. שיטת הניקוד מותאמת דינמית לקלט של המשתמש כדי להבטיח רלוונטיות גבוהה עוד לפני שלב ה-AI.

**שלב 2 — AI:**
ה-AI (Gemini) מקבל רק את 10 הפריטים שעברו את הסינון, בוחר מתוכם את 5 המתאימים ביותר וכותב הסבר אישי לכל אחד. ככה המערכת לא שולחת את כל המאגר ל-AI בכל בקשה — מה שמוריד עלויות ומאיץ את זמן התגובה.

---

## הוראות הרצה

הורד את הפרויקט למחשב.

בתיקיית `backend`, צור קובץ `.env` והוסף את המפתח שלך:
```
GEMINI_API_KEY=your_api_key
```

הרצת השרת:
```bash
cd backend
npm install
npm start
```

הרצת הממשק:
פתח את `frontend/index.html` בדפדפן.

הרצת הטסטים:
```bash
cd backend
npm test
```

## החלטות עיצוב מרכזיות

**שיטת הניקוד (Scoring):**
שיטת הניקוד מבוססת על משקלות משתנות, המאפשרות לתת עדיפות גבוהה יותר למאפיינים רלוונטיים (כמו אירוע או סגנון) בהתאם למה שהמשתמש ציין, מה שמבטיח דירוג מבוסס נתונים וצפוי.

**חלוקת תפקידים ברורה:**
הלוגיקה העסקית נשארת אצלי, ה-AI אחראי רק על הניסוח וההסבר האישי. זה מאפשר שליטה מלאה על התוצאות ומונע מצב שה-AI ממציא פריטים שלא קיימים.

**Middleware:**
הוספתי שכבת validation שבודקת שגיל ותקציב הם ערכים הגיוניים לפני שהבקשה מגיעה ל-controller, ו-error handler מרכזי שתופס שגיאות בכל שכבה ומחזיר תשובה מסודרת.

**Fallback:**
אם ה-AI לא זמין מכל סיבה, המערכת לא נופלת — היא מחזירה את 5 הפריטים עם הניקוד הגבוה ביותר עם הסבר אוטומטי שנבנה מהנתונים.

---

## שימוש ב-AI

השתמשתי ב-Gemini 2.5 Flash. במהלך הפיתוח, אחת ההחלטות החשובות שקיבלתי נגעה לאופן שבו אני משלבת AI במערכת. בתחילה עלתה האפשרות לשלוח ל-LLM את כל נתוני המאגר כדי שידרג בעצמו את התוצאות, אך הבנתי שגישה זו תהיה יקרה, איטית מדי (Latency), ותבזבז משאבים יקרים על עיבוד של מידע לא רלוונטי.

כדי לייעל את המערכת, בחרתי בגישת **Hybrid Filter-then-Rank**:

**שלב הסינון (Deterministic):** הפעלתי לוגיקה קשיחה שרצה על בסיס הנתונים ומסננת מראש רק את הפריטים שעומדים בתנאים הבסיסיים (כמו תקציב והעדפות צבע). שלב זה מהיר מאוד ואינו עולה דבר.

**שלב הדירוג וההסבר (LLM):** רק לאחר הסינון, אני שולחת את רשימת המועמדים המצומצמת ל-Gemini. כך, ה-AI מתמקד רק במשימה שבה הוא מצטיין — הוספת הטאץ' האישי והסברים מנוסחים היטב — מבלי לעבד מידע עודף.

גישה זו מאפשרת לשלוט בעלויות ה-API בצורה מושכלת, להבטיח זמני תגובה מהירים למשתמש, ולשמור על המערכת יציבה וחסכונית.

## טיפול במקרי קצה

חשוב לי שהמערכת תתנהג בצורה צפויה גם כשהקלט לא מושלם:

- **פרופיל חסר שדות** — כל שדה אופציונלי. אם לא סופק occasion, המשקלות מתאימות את עצמן אוטומטית. אם לא סופק תקציב, כל הפריטים עוברים סינון.
- **אין פריטים מתאימים** — המערכת מחזירה מערך ריק עם הודעה ברורה, ולא קורסת.
- **AI לא זמין** — קיים fallback אוטומטי שמחזיר תוצאות מבוססות ניקוד עם הסבר שנבנה מהנתונים.
- **קלט לא תקין** — middleware של validation תופס ערכים בלתי הגיוניים (תקציב שלילי, גיל מעל 120) ומחזיר שגיאה ברורה לפני שהבקשה מגיעה ל-controller.

## שאלת סקייל

אם היו פה 100,000 פריטים ומיליוני משתמשים:

- **מסד נתונים:** הייתי עובר ל-Vector DB (כמו Pinecone) לחיפוש סמנטי מהיר במקום לסרוק את כל המאגר בכל בקשה.
- **Cache:** הייתי משתמש ב-Redis לשמירת תוצאות נפוצות, כדי לא לחשב מחדש המלצות זהות.
- **תורים:** חישוב ההמלצות היה רץ ברקע (Queue) כדי שהמשתמש לא יחכה בעומס.
- **Rate limiting:** הגבלת קריאות ל-AI כדי לשלוט בעלויות.
